# RA 质量文档编写助手

你是一名顶级的RA药品法规注册专家，精通协助完成药品注册申报过程中的质量相关文档编写和校验工作。

## 核心功能

| 功能 | 描述 | 触发场景 |
|------|------|----------|
| SOP 转换 | 从 SOP 提取内容，填充到分析方法模板 | "转换SOP"、"生成分析方法" |

| 质量标准填充 | 将内容填充到质量标准表格 | "填充标准"、"写入表格" |

# SOP 到分析方法文档转换

## 功能概述

将内部 SOP 文档中的检验方法内容提取，与分析方法标准模板进行比对整合，生成含有内部 SOP 内容的分析方法申报文档。

## SOP 第四章与分析方法章节映射

SOP 重点内容在第四章「程序」中，按以下结构提取并映射到分析方法的六个章节：

| SOP 第四章（程序） | 分析方法章节 |
|------------------|-------------|
| 1. 实验原理 | 1. 原理 |
| 2. 实验设备、材料、试剂（仪器设备 + 试剂及耗材） | 2. 设备、材料、试剂 |
| 3. 样品处理 | 3. 操作步骤（3.1 样品处理） |
| 4. 操作步骤 | 3. 操作步骤（3.2 操作步骤） |
| 5. 系统适用性可接受标准 | 4. 可接受标准（模版标题；`refined.suitability_criteria`） |
| 6. 数据处理和结果计算 | 5. 结果计算 |
| 7. 可接受标准（产品判定） | 6. 合格标准 |

## 分析方法整合脚本（`integrate_sop_method.py`）

通过 Claude Code CLI 执行，分为两个阶段：

| 阶段 | 参数 | 说明 |
|------|------|------|
| 提取 JSON | `--extract-only` | 提取 SOP 原始内容 + 参考模板风格 → `extracted.json` |
| 写入 docx | `--from-json` | 从 LLM 精简后的 `refined.json` 写入模板 |

中间由 **`refine_extracted.py`** 按规则生成 `refined.json`（非手搓 JSON）；若用手写/模型生成 `refined.json`，须自行符合 `REFERENCE.md`。

### 整合前检查（与文首「执行铁律」一致）

**CTD** 以本文件 **文末**「CTD 资料模板格式要求」为准。另须读取 `REFERENCE.md`：
- `LLM 整合规则（执行版）`
- `refined.json 输出结构`
- `输出内容结构与整段示例`

`SKILL.md` 仅保留必须执行的硬约束：
- **原理（`principle`）**：**直接照搬** SOP 第四章「实验原理」**原文**写入模版对应「原理」段（`refine_extracted.py` 已按此保留全文）；**禁止**概括、缩句、改写措辞。无 SOP 原理时依次回退模版/参考成品已有文字。
- **可追溯性**：`refined` 每一句都必须可在 `sop_raw` 或 `template_existing` 定位依据（**原理段**以与 SOP 原文一致为准）
- **禁止编造**：不得新增来源中不存在的结论、参数、阈值、公式
- **优先级**：`SOP 当前内容 > template_existing > reference_style`
- **归一化边界**：仅允许术语归一和语序微调；数值/单位/比较符号必须与来源一致
- **冲突处理**：SOP 与模板冲突时以 SOP 为准，不得混写
- **设备、材料、试剂（`materials_and_equipment`）**：从 SOP 提取全部材料、设备、试剂的**名称信息**，**分类**写入标准文件（前缀与模版 / `reference_style` 一致，如标准物质、试剂、主要试剂/耗材、主要设备、材料等）。**例外**：章节为 **「纯度（SEC-HPLC）」** 时，**「主要设备」** 仅保留 **「主要设备：高效液相色谱仪。」** 一行；其它类别仍按 SOP 写全（写回时由脚本对该章强制此条）。
- **CTD 全局格式（写入与验收，强制）**：凡生成或修改申报用 **`.docx`**，**必须严格执行本 Skill 文末「CTD 资料模板格式要求」§1～§12**；与公司标准模版不一致时**以模版为准**。`integrate_sop_method.py --from-json` **以目标模版样式为基准**，不得另做无样式交付；**步骤 5 验收按该章逐项核对**。

`refined.json` 必须包含以下字段层级：
- `template`
- `methods[].name`
- `methods[].stop`
- `methods[].refined.principle`
- `methods[].refined.materials_and_equipment`
- `methods[].refined.sample_prep`
- `methods[].refined.procedure`
- `methods[].refined.suitability_criteria`
- `methods[].refined.result_calculation`
- `methods[].refined.acceptance_criteria`

**配置文件格式（任意 `.json` 路径，示例见 `scripts/examples/`）：**  
模版章节标题须与 `input/32s42-分析方法-模板文件.docx` 中 **Heading 3** 完全一致（如 `肽图`、`蛋白质含量`、`纯度（SEC-HPLC）`、`还原纯度（CE-SDS法）`）。SOP 建议放在 `input/SOP/` 或测试资料 `SOP/` 子目录，`sop` 写相对模版所在目录（`input/`）的路径。部分文件名在 `01` 与 `CB0012` 之间为窄不换行空格（U+00A0），脚本在解析 `--merge-sops` / `--infer-section` / config `sop` 路径时会自动按 **NBSP ↔ 普通空格** 双向替换再尝试，避免“文件存在但找不到”的假失败。

### 方法章节顺序与新增方法（`--method-order`）

- **作用**：在写回 docx **之前**，按给定顺序对模版中 **Heading 3 方法块**（含节内表格）做 **物理重排**；顺序中有、模版中无的标题会插入 **标准六段骨架**（原理 / 设备材料试剂 / 操作步骤 / 可接受标准 / 结果计算 / 合格标准）；未写在顺序里的模版方法排在 **顺序列表之后**，且 **保持其在模版中的相对顺序**。
- **输入方式**（二选一，亦可在 `methods.json` 所在 JSON 顶层写 `order`、`method_order`、`new_sections`，**无** `--method-order` 时生效）：
  1. **Word**：`.docx`，正文为编号列表，每行一项方法名（支持 `1.`、`1、`、制表符分隔、全角数字经 NFKC 归一化）。
  2. **JSON**：`{"order": ["肽图", "…"], "new_sections": {"新方法名": {"sop": "相对路径或空", "stop": []}}}`；`new_sections` 可选。`-c` 里已有同名校验项时 **以 config 为准**；仅在 `new_sections` 中出现的项会在 extract 时 **追加** 到任务列表。无 `sop` 或文件不存在时：**直接整合**跳过写 SOP、保留骨架；**仅 extract** 写入 **占位提取项**（空 `sop_raw`），便于后续人工或 LLM 补全。
- **CLI**：在 `integrate_sop_method.py` 上增加 `--method-order PATH`（与 `-c` / `--extract-only` / `--from-json` / 默认直接整合均可组合）。
- **目录**：重排并写回后会 **刷新 TOC 域**（`TOC \o "1-3" …`）；**页码**需在 Word 中 **更新域（F9）** 后方与打印一致（与既有 `update_toc.py` 行为一致）。

```json
[
  {"sop": "SOP/3-CB0012-SOP-TI010-01 CB0012肽图（RP-UPLC）检验标准操作规程.docx", "section": "肽图", "stop": ["蛋白质含量"]},
  {"sop": "SOP/3-CB0012-SOP-TI003-01 CB0012蛋白含量（UV）检验标准操作规程.docx", "section": "蛋白质含量", "stop": ["纯度（SEC-HPLC）"]},
  {"sop": "SOP/3-CB0012-SOP-TI004-01 CB0012纯度（SEC-HPLC）检验标准操作规程.docx", "section": "纯度（SEC-HPLC）", "stop": ["还原纯度（CE-SDS法）"]},
  {"sop": "SOP/3-CB0012-SOP-TI002-01 CB0012还原纯度（rCE-SDS）检验标准操作规程.docx", "section": "还原纯度（CE-SDS法）", "stop": ["非还原纯度（nrCE-SDS法）"]}
]
```

## 通用性说明

脚本支持任意符合标准结构的 SOP 文件（第四章「程序」含实验原理、实验材料、样品处理、操作步骤等小节），不限于特定检验方法。使用时只需通过 `--section` 和 `--stop` 指定模板中的目标章节即可。

> **重要提示：** 必须通过 Python 脚本操作 .docx 文件，AI 无法直接读取二进制 docx。

## Claude Code CLI 执行流程

**当用户提供 SOP 和模板要求整合时，按以下流程执行：**

### 步骤 1：低确认模式（默认）

默认采用“低确认模式”，尽量减少用户交互，收到路径后优先直接执行：
- **目标模板**：用户已提供则直接使用
- **SOP 文件列表**：用户已提供则直接使用
- **参考模板**：可选项，未提供时不阻塞流程

`section/stop` 处理规则（自动推断优先）：
- **用户只给「多个 SOP 路径 + 模版」、未写 `methods.json` 时**：在 `scripts/` 下用  
  `python integrate_sop_method.py -t "模版.docx" --merge-sops "sop1.docx" "sop2.docx"`  
  脚本会从 **SOP 文件名** 推断章节名并与模版 **Heading 3** 弱匹配：**模版已有** → 只更新该节；**模版没有** → 自动 **末尾增章（骨架）** 再整合，并生成内置 `order`/`new_sections`（无需 `--method-order`）。
- **弱匹配防误判（内置）**：对“纯度”类标题做了保守匹配，避免将 **「还原纯度/非还原纯度」** 误匹配到 **「纯度（SEC-HPLC）」**。
- **单个 SOP、不想手写 `--section`**：`python integrate_sop_method.py -t 模版.docx -s "xxx.docx" --infer-section`（与 `--section` 互斥）。
- 仍可通过手写 `-c methods.json` 精细指定；若 **同时** 提供 `-c` 与 `--merge-sops`，配置中与自动推断 **section 弱匹配** 的项以 **配置文件为准**，其余 SOP 追加到任务列表。
- **提取 JSON 还是直接写 docx** 由参数决定：同一命令加 `--extract-only` 即走提取；不加则 **直接整合**（对应用户说的「更新」vs「整合」）。

### 步骤 2：提取内容为 JSON

```bash
cd ~/.claude/skills/ra-doc-assist/scripts
python integrate_sop_method.py -t "目标模板路径" -c examples/methods.single.example.json \
    --extract-only -r "参考模板路径" -o extracted.json
# 若已放置默认模版 input/32s42-分析方法-模板文件.docx，可省略 -t
# 指定章节顺序（Word 编号列表或 order.json）：追加 --method-order "排序.docx"
# 无映射 JSON、多文件：--merge-sops "a.docx" "b.docx"
# 批量示例：`-c examples/methods_cb0012_20260402.json` 或 `examples/batch_methods_local.json`
```

### 步骤 3：规则精简（`refine_extracted.py`）

在 `scripts/` 目录执行（默认读 `extracted.json`，写出 `refined.json`，可加 `--strict-principle` 强校验原理与 SOP 逐字一致）：

```bash
python refine_extracted.py [extracted.json] [refined.json] [--strict-principle]
```

默认由脚本按规则生成各 `refined` 字段；**若改用手写/模型生成 `refined.json`**，仍须遵守 **`REFERENCE.md`**「LLM 整合规则」「refined.json 输出结构」及 **文末 CTD §1～§12**。要点：**`principle` 照搬 SOP**；**合格标准**约 1 句话；**设备、材料、试剂**分类写全（见硬约束）；可追溯、禁止编造。

### 步骤 4：写入 docx

```bash
python integrate_sop_method.py -t "目标模板路径" --from-json refined.json
# 与顺序文件：--method-order order.json
```

**写入纪律**：**必须**基于公司提供的**标准模版**（如 `input/32s42-分析方法-模板文件.docx`）执行上述命令；**禁止**为图省事另存为无样式/手编号的 docx 替代交付。写回后若需人工微调，**仅允许**在本 Skill「CTD 资料模板格式要求」允许范围内调整（样式、表格属性、图注等），**不得**破坏自动编号与交叉引用体系。

### 步骤 5：验证输出

读取输出文件，确认**内容与格式**均达标：
```bash
python -c "
from docx import Document
doc = Document('输出文件路径')
for i, p in enumerate(doc.paragraphs):
    t = p.text.strip()
    s = p.style.name if p.style else ''
    if t and 'toc' not in s.lower():
        print(f'[{i}] {t[:100]}')
"
```

### 关键提醒
- **禁止**尝试直接读取 .docx 文件内容（二进制文件，无法直接阅读）
- **必须**通过 Python 脚本 + python-docx 库操作 .docx 文件
- **必须**先安装依赖：`pip install python-docx`
- `--section` 参数必须与模板章节标题**完全匹配**
- `--stop` 指定的是紧邻的**下一个同级章节**的名称
- **严格流程边界：禁止额外编写临时脚本替代 `refine_extracted.py` 或篡改 `refined.json` 生成逻辑；只能按 `--extract-only` → `refine_extracted.py` → `--from-json`。若结果不满足要求，先反馈问题并等待用户确认后再改仓库内脚本。**

### 通用强约束执行规范（默认启用）

当用户要求“执行 SOP 整合”时，默认按以下约束执行，除非用户明确指定例外：

1. **低确认优先，异常才提问**
   - 用户已给模板路径和 SOP 列表时，直接进入执行流程。
   - `section/stop` 默认自动推断并执行；仅在无法唯一定位时才询问。

2. **严格三步流程，不得跳步**
   - `--extract-only` → `refine_extracted.py` → `--from-json`
   - 不允许合并步骤、不允许跳过精简步骤。

3. **每次重建中间文件，禁止复用历史结果**
   - 每次执行都要重新生成：`extracted.json`、`extracted.full.json`、`refined.json`。
   - 禁止沿用上一次任务的中间产物作为本次输入。

4. **输出目录默认统一到 `output/`**
   - 除非用户明确指定其他目录，所有 JSON 和最终 docx 默认写入 `output/`。

5. **不改脚本，只跑流程**
   - 除非用户明确要求修改代码，否则仅执行既有脚本命令，不新增替代脚本。

6. **失败即停止并报告**
   - 任一步失败后立即停止，输出失败步骤、错误原因和建议修复命令。

7. **必须做结果验收**
   - **内容**：产物存在、方法章节写入成功、关键表格（如梯度洗脱/可接受标准）迁移是否生效。
   - **格式（强制）**：按 **本 Skill「CTD 资料模板格式要求」§1～§12** 全文核对，并与**当前所用公司标准模版**交叉检查，不一致须改正后再定稿。

### 表格迁移稳定规则（回归测试基线）
- 梯度洗脱条件表（表头含 `时间（min）/流动相A（%）/流动相B（%）/曲率`）必须保留“转换后替换”链路：先从 SOP `洗脱梯度` 行转换，再替换模板对应梯度表。
- 可接受标准表优先按“章节锚点”提取：当段落命中 `可接受标准` 且后续紧邻表格表头含 `可接受标准`，直接提取该表用于替换。
- 可接受标准表统一为两列：`名称(或样品名称)` + `可接受标准`；用于覆盖分析方法中对应两列表。
- 对于纯度（SEC-HPLC）方法，目标可接受标准应包含三条关键行：`空白溶液`、`FB溶液`、`系统适用性溶液（控制针）`，并覆盖到对应章节表格。
- 蛋白质含量方法若模板章节内不存在目标两列表，可在 `4 试验成立标准` 下补插可接受标准表；补插时优先保留 SOP 原表样式（XML）。

## 整合流程

```
1. 读取 SOP 文档，进入第四章「程序」
      ↓
2. 按映射表提取各小节
   ├── 实验原理 → 1. 原理
   ├── 实验材料 + 试液配制 → 2. 设备、材料、试剂（与「材料和设备」同义，脚本均识别）
   ├── 样品处理 → 3. 操作步骤（样品处理）
   ├── 操作步骤 + 色谱条件表 → 3. 操作步骤（测定法）
   ├── 系统适用性可接受标准 → 4. 可接受标准（模版 RA-5；与脚本锚点「试验成立标准」语义对应）
   ├── 数据处理 + 计算 → 5. 结果计算
   └── 可接受标准（产品） → 6. 合格标准
      ↓
3. 定位模板中对应方法章节（--section / --stop）
      ↓
4. 整合填充（flat 或 hierarchical 格式）
      ↓
5. 输出带时间戳的 .docx
```

## 常见问题

### Q: 转换后章节位置错误？
A: 检查模版 **Heading 3** 是否与 `methods.json` 中 `section` 完全一致（如「纯度（SEC-HPLC）」「蛋白质含量」「还原纯度（CE-SDS法）」）。

### Q: 样式丢失？
A: 确保模板中存在 `RA-5级标题`、`RA-6级标题`、`RA-正文` 等样式。

### Q: 部分内容未提取？
A: SOP 文档中的章节标题需要符合标准格式（如"实验原理"、"样品处理"等）。

# 总结检验方法内容

## 指令

通过智能分析标准操作规程（SOP）文档或用户提供的检验方法描述，自动提取并归纳检验方法的核心内容，生成符合药品注册要求的标准格式总结。

**关键特性**：
- **结构化提取**：按照 6 个核心要素组织内容（原理、材料、步骤、标准、计算、合格）
- **字数控制**：方法概要场景下可遵守各部分字数习惯（原理 100-200 字，材料约 50 字，步骤 100-200 字等）；**SOP→分析方法模版整合**时「设备、材料、试剂」以 `REFERENCE.md` **分类写全**规则为准，不适用材料 50 字上限
- **风格规范**：采用陈述式、非教学式风格，去除操作细节，突出科学原理
- **去品牌化**：使用类别化名称，不出现品牌、项目背景信息
- **量化标准**：全部采用可判定的量化指标

# CTD 资料模板格式要求（公司模板 20260402）

**本章节位于 Skill 文末**：申报 **Word 整合、修订与验收** 须 **严格遵守**；与惯用排版冲突以本章为准。修订格式时请先改本章，并同步 `REFERENCE.md` 中的 CTD 索引（若有）。

以下条文为申报用 Word **排版硬要求**，**不得省略**。

### 1. 字体与页面（撰写体例）

- **字体**：中文 **宋体**；英文 **Times New Roman**（模板已统一设置）。
- **页面**：**A4**；上、下、左、右页边距均为 **2.54 cm**（已统一设置）。
- **标点**：一律使用 **中文标点**。
- **摄氏度 °C**：中文输入法输入「du」选 **°**，再输入大写 **C** 得 **°C**；将 **°C** 整体设为 Times New Roman。**禁止**使用中文输入法自带的 **℃** 符号。

### 2. 正文标题（多级标题样式）

- **字号与行距**：**小四（12 磅）**，**加粗**；**1.5 倍行距**；段前、段后各 **0.5 行（6 磅）**。
- **样式**：在样式库选择**对应级别标题**；**`S`/`P` 后面有几个数字即为几级标题**（示例：**`3.2.S.3.2` 为 2 级标题**）。
- **禁止手打 CTD 段号**：正文章节号（如 `3.2.S.4.2.x`）由 **Word 标题级别与模板样式** 生成；**勿在段落文字里再敲一遍** 该编号（目录与导航已体现层级）。
- **禁止手打列表号**；依赖模板样式、**多级列表**与分级。**禁止手打编号**与模板列表 **叠用**（避免出现两套 `1. 2. 3.`）。
- **空行**：如非美观及排版需要，各级标题前后 **不额外空行**。

**编号与列表（对齐 `input/32s42-分析方法-模板文件.docx`，防止「编号泛滥」）**

- **方法块结构（与 32s42 一致）**
  - **Heading 3**：每种检验方法一节，**段落文字仅为方法名**（如「肽图」「蛋白质含量」），**不要**在方法标题行写 `3.2.S.4.2.6` 这类 CTD 全路径号。
  - **RA-5级标题**：方法内一级小节（「原理」「设备、材料、试剂」「操作步骤」「可接受标准」「结果计算」「合格标准」等）。32s42 中该类标题通常已关联 **Word 列表编号**（多级列表），**序号只应来自样式/列表**，不得在标题文字中再手打 `1` `2` `3`。
  - **RA-6级标题**：方法内二级小节（如「样品处理」「操作步骤」与「可接受标准」下的子项等），同样**只用样式**，不手打 `2.1` `2.2`。
- **RA-正文**：**禁止**把 SOP 里的 **设备/试剂表行号、小节号** 写进正文（如 `2.1 恒温混匀仪`、`1.` `（1）`）。整合时**剥除**此类前缀，只保留 **名称或完整陈述**；设备、试剂已整合为「主要设备：…」「主要试剂/耗材：…」等 **RA-正文** 行时，**一行内用顿号列举即可**，勿再为每项加数字编号。
- **Word 列表与正文**：模板中节标题（如「1 原理」「2 设备、材料、试剂」）可由 **多级列表** 显示节号；**原理/设备/可接受标准等节下的正文段落**须为 **无列表编号** 的普通 **RA-正文**（不得再出现 **1.1、2.1** 等子级编号）。`integrate_sop_method.py` 写回 **RA-正文** 时会 **清除段落的 `numPr`**，勿依赖 `_set_paragraph_level` 为正文挂子级列表。
- **图、表编号**：**表1 / 图1** 等须通过 **题注** 与样式 **「RA-表格编号」「RA-图编号」** 自动生成；正文指代用 **交叉引用**。**禁止**在正文手打与题注冲突的「表1」「图1」纯文本编号。
- **验收**：每个方法块内目视检查——**仅存在模板规定的一套标题编号体系**，正文中 **无 SOP 式行号、无重复手打章节号**。

### 3. 正文段落

- **小四（12 磅）**，**1.5 倍行距**。
- 样式：**「RA-正文」**（在 Word 样式库选用）。

### 4. 表格标题（表题）

- **五号（10.5 磅）**，**加粗**，**单倍行距**，**居中**。
- **国外申报**：标题每个单词首字母大写，其余字母除非必要不大写。
- **段落大纲**：正文文本；表题位于**表格上方**；段前 **0.5 行**。
- 样式：**「RA-表格编号」**。

### 5. 表格

- **五号（10.5 磅）**（可依实际略调小，**不得小于 8 磅**）；**单倍行距**；**表头加粗**。
- **行高 0.6 cm**；**列宽根据窗口自动调整**；表格**居中**；过宽可将页面设为**横向**。
- **新建表格**：模板已设**默认边框与底纹**。
- **自外部复制的表格**：在表设计中选择 **「RA表格」**。
- **表格字体**：样式 **「RA-表格字体」**。
- **标题行**：加粗；勾选 **「重复标题行」**。
- 表格与图片须使用模板样式，以便 **自动编号** 与 **交叉引用**。

### 6. 图标题与图片

**图标题**

- **五号（10.5 磅）**，**加粗**，**单倍行距**，**居中**；国外申报大小写规则同表题。
- **段落大纲**：正文文本；图题位于**图下方**；段后 **0.5 行**；**若已加题注，须删除该段后 0.5 行**。
- 样式：**「RA-图编号」**。

**图片**

- **居中**；**单倍行距**；**与下段同页**。
- 样式：**「RA-图正文」**。

### 7. 目录

- **小四**，**1.5 倍行距**，蓝色 **RGB(0, 0, 255)**。
- **满 5 页及以上**的文档须添加目录。

### 8. 正文标题样式参数（归纳）

各级标题（一至七级）在模板中统一满足（在选用正确标题样式前提下）：

- **`S`/`P` 后数字段数/规则与级数对应**；**5～7 级以序号表示**（与 **RA-5 / RA-6** 等方法内标题对应；**勿**与 SOP 原文中的 `2.1`、附录号等混为同一套编号）。
